# Pipeline UI subsystem

> Moved verbatim from the plugin `CLAUDE.md` — load-bearing contracts, loaded on demand. Follow each bullet's lockstep rules and bump the plugin `version` in `plugin.json` on any meaningful change (see `CLAUDE.md`).

Lives under `apps/pipeline-ui/` and `hooks/{pipeline_ui_relay,analytics_relay}.ts`. **Write scope with respect to the consumer project: STRICTLY inside `<project>/.claude/pipeline/`** — historically the daemon was fully read-only; since the launcher/editor refactor it may also (a) write pipeline files under `.claude/pipeline/**` through the guarded editor endpoints (`editor.ts` — path-traversal-proof, extension allow-list, `.runtime`/`.stats`/`.feedback` forbidden, optimistic concurrency), and (b) spawn `pipeline drive` runs whose artifacts land under `<pipeline_root>/.runtime/<run-id>/` (`launcher.ts` — `pipeline_root` validated to resolve inside the project's pipelines dir before anything is spawned or written). It must NEVER touch consumer files outside `.claude/pipeline/`. Reading is unchanged: `.runtime/events.jsonl` (written by `/pipeline:run` and the hooks) and the pipeline manifests.

Two sibling modules extend `server.ts` (same decoupling contract as `transcripts.ts` — a `getProject` callback, no registry sharing):

- **`launcher.ts` — the run launcher.** `GET /api/pipelines` (computePlan-backed catalog: steps + resolved models for the per-step override UI), `POST /api/runs/launch` (spawn `pipeline drive` with `--task`/`--task-file`/`--model` flags; cwd = project root; `PIPELINE_UI_ENABLED` inherited so events light the dashboard), `POST /api/runs/answer` (deliver the answer to a parked needs-input run — the SAME executor session resumes), `GET /api/drive-runs` (snapshots incl. the question). Exit code 4 = awaiting-input; every state change broadcasts `drive.run` over SSE; `launch.json` + `drive.log` persist under `<pipeline_root>/.runtime/<run-id>/` so answers survive a daemon restart. `findDriveUsage` feeds `/api/run-stats` for headless runs (no transcript binding exists — the envelope-usage fold in `usage.json` provides tokens + `cost_usd` instead of zeros). `POST /api/runs/stop` (server.ts route) is the user-facing cancel: `stopDriveRun` kills the live drive child (if this daemon owns one) and finalizes its snapshot as halted ("stopped by user"), then the route appends the synthetic `pipeline.halted` journal event in every case — so it also clears stale manager-driven runs stuck on "running".
- **`editor.ts` — the pipeline editor.** `GET /api/editor/list|file`, `PUT|DELETE /api/editor/file`, `POST /api/editor/create-step` (designer-template `NN-*.md` scaffold), `POST /api/editor/validate` (computePlan lint). Deleting `PIPELINE.md` is refused. Every write invalidates the pipeline/catalog caches and broadcasts `file.changed`. The web editor edits frontmatter through a ROUND-TRIP-SAFE form (`web/src/lib/editorFrontmatter.ts` keeps unknown keys byte-for-byte) — keep that property when adding fields.
- **`aifix.ts` — AI Fix.** `POST /api/editor/ai-fix` hands the validate lint issues to a background `claude -p --model <haiku|sonnet|opus|fable> --permission-mode acceptEdits` session (cwd = project root, prompt instructs edits ONLY inside the pipeline root; same containment check as the launcher). One job per pipeline at a time; 10-minute kill timeout; the UI polls the job snapshot via `GET /api/editor/ai-fix?job_id=` (no SSE channel — a rare user-triggered job doesn't warrant one). The web editor shows the button + model picker under the validate results, ticks an elapsed timer while running, then re-validates and reloads on completion.
- **`transcribe.ts` — the speech-to-text proxy.** `GET /api/transcribe/status` + `POST /api/transcribe` forward browser-recorded audio to an OpenAI-compatible Whisper endpoint (provider by env: `PIPELINE_STT_URL[_KEY/_MODEL]` custom ▸ `OPENAI_API_KEY` ▸ `GROQ_API_KEY`; `PIPELINE_STT_PROVIDER` forces one). Keys stay server-side; audio goes ONLY to the configured provider and is never persisted. `?lang=auto` (the web default) sends NO language hint — Whisper auto-detects, which is what makes MIXED Russian+English dictation in one sentence come out right; a pinned `?lang=ru|en` still forces one. No provider → `{available:false}` and the web `VoiceInput` falls back to the browser's own Web Speech API (single-language only).

**Host binding + token auth (`resolveHostConfig`/`checkAuth` in `launcher.ts`).** The daemon binds `127.0.0.1` by default. `PIPELINE_UI_HOST` widens the bind (phone access), but then `PIPELINE_UI_TOKEN` is MANDATORY — the UI can launch runs and edit files, so a non-loopback bind without a token falls back to loopback with a loud boot warning. With a token set, every request must carry it (`Authorization: Bearer`, `?token=`, or the cookie a first `?token=` page-load pins). Do not weaken this: the launch/editor endpoints are the reason the gate exists.

Three layers, each with a single owner:

| Where | Who writes | What |
|---|---|---|
| `${CLAUDE_PLUGIN_ROOT}/apps/pipeline-ui/` | this plugin (read-only at runtime) | daemon code + pre-built React bundle |
| `~/.claude/pipeline-ui/` | the daemon + hooks | `daemon.lock` (pid+port), `projects.json` (registry), `active-mirror-bindings.jsonl` (issue #11 — append-only mirror bindings), log files |
| `<project>/.claude/pipeline/.runtime/` | `/pipeline:run` + hooks + daemon | `events.jsonl` (append-only journal), `chat-messages.jsonl` (Agent-SDK + mirrored chat rows), `transcripts/<session>.offset` (token-counter cursors), `transcripts/<session>.chat.offset` (mirror-tail byte cursors) |

This split is load-bearing: the plugin install dir must stay read-only (see "Path resolution rules" in the plugin `CLAUDE.md`), but the daemon needs per-user state, and per-project events must travel with the project (gitignored).

## Hard invariants

- **The daemon is shared across all projects on the machine.** One process, one port, one URL. Project switching happens inside the browser, not by spawning multiple daemons.
- **Worktrees resolve to their main repo.** A git worktree is NOT a separate project for UI purposes — its events land in the main repo's `.runtime/events.jsonl` with a `worktree:` tag on each event. Worktree detection logic in `apps/pipeline-cli/src/lib/event.ts` (the TS event writer `pipeline event` calls), `analytics_relay.ts`, and `pipeline_ui_relay.ts` MUST stay in sync. If you change one, change the others — they all read `.git/commondir` the same way.
- **The run anchor is `pipeline-manager`; the worker is `step-executor` (legacy `pipeline-executor` tolerated).** As of the Phase-2 UI rekey, the run is identified by the `pipeline-manager` spawn, not the per-step worker. The hook matches the manager with `MANAGER_SUBAGENT_RE = /^(?:[a-z0-9_-]+:)?pipeline-manager$/` and the worker with `WORKER_SUBAGENT_RE = /^(?:[a-z0-9_-]+:)?(?:step-executor|pipeline-executor(?:-(haiku|sonnet|opus))?)$/` (the legacy worker name + its long-removed per-tier suffix stay tolerated so in-flight/forked runs still bind). A worker spawn is **never** a run anchor — it only gets a mirror binding attributed to the owning run.
- **`/pipeline:run` is the canonical semantic emitter.** Hooks emit only what they're naturally positioned to know: `SessionStart` → `session.opened`; `PreToolUse` (manager Agent/Task spawn) → mirror binding write (issue #11) **+ RUN-LEVEL bypass START-half synthesis** (`pipeline.started` ONLY); `PostToolUse` → `tool.called` + RUN-LEVEL bypass **END-half synthesis** (`pipeline.completed`/`halted` ONLY), or the both-event fallback when `PreToolUse` never ran; `SubagentStop` (a `pipeline-manager` ends) → `manager.stopped`; `Stop` → `turn.usage`. The `SubagentStop` hook is a **liveness signal**, NOT a re-derivation of the manager report — `/pipeline:run` already parses the report; emitting lifecycle from `manager.stopped` would be double-emission. The hooks NEVER synthesize `iteration.*` — the `pipeline-manager` self-emits those.
- **Bypass (Path-C) lifecycle synthesis is RUN-LEVEL, anchored on the manager, and SPLIT across PreToolUse/PostToolUse — keep it split.** The whole point is that a directly-spawned `pipeline-manager` (no `/pipeline:run`) shows up as an **active** run while the manager is in-flight — so `pipeline.started` MUST be emitted at spawn time (`PreToolUse`), not deferred to the Agent's return; `pipeline.completed`/`halted` is the END half (`PostToolUse`). Both halves share one run_id = `sha1(tool_use_id).slice(0,12)`. Path B is told apart by (1) the literal `run_id = …` line the supervisor writes into the manager prompt (`extractRunIdFromPrompt`), and (2) failing that, comparing our tool_use_id-derived id against the journal's most-recent matching `pipeline.started`/`iteration.started.run_id` (different id ⇒ supervisor owns it ⇒ stay silent). `journalHasPipelineStarted(runId)` chooses split-END vs both-event fallback in `PostToolUse`. If you change the split, the run_id derivation, the manager/worker regexes, or the discriminator, update `synthesizeBypassStart`/`synthesizeBypassEnd`/`synthesizeBypassRun` in `hooks/analytics_relay.ts`, `EVENTS.md` (Path-C section), and `tests/hook-pretool-binding.test.ts` + `tests/hook-bypass-detection.test.ts` in lockstep. A Path-C run killed between the two halves leaves a `pipeline.started` with no terminal event; the **primary** clear is the `manager.stopped`-driven sweep (next bullet), with the Dismiss button as the guaranteed fallback (the hooks write no liveness lockfile).
- **Liveness has a NEW primary signal: `manager.stopped` via SubagentStop.** When a `pipeline-manager` subagent ends, the SubagentStop hook emits `manager.stopped { run_id, agent_id }` (run_id resolved by the same session-keyed binding recovery as `tool.called`). The daemon's `sweepManagerStoppedRuns` (in `server.ts`, run alongside `sweepProjectLiveness`) marks a run abandoned (`pipeline.halted`, `abandoned:true`) when it has a `pipeline.started` and a `manager.stopped` but no terminal event — UNLESS a live `<run_id>.alive` lockfile still names a running driver (the Path-B supervisor poll-waiting on a nested blocker, which legitimately stops + re-spawns the manager). The **pid-lockfile machinery (`pipeline event write-liveness`/`clear-liveness` + `sweepProjectLiveness`) is KEPT as a deliberate secondary fallback** — do NOT remove it. `manager.stopped` is a new event TYPE, not a schema-version bump (the daemon tolerates unknown types). If you change the manager-stop contract, update `handleSubagentStop`/`hooks.json` (SubagentStop registration), `sweepManagerStoppedRuns` in `server.ts`, `EVENTS.md`, and `tests/hook-subagent-stop.test.ts` + `tests/server.test.ts` in lockstep. **The `SubagentStop` registration carries `"matcher": "pipeline-manager"`** — `SubagentStop` is one of the few events that supports a matcher (it filters on `agent_type`; `Stop` does NOT support one), and the value is a regex substring (the hyphen forces regex mode) that matches both the local `pipeline-manager` and the marketplace-namespaced `pipeline:pipeline-manager`, while excluding every other subagent (`step-executor`, `pipeline-improver`, …) so the hook stops spawning a Bun process for non-manager subagent stops. This narrowing is a pure perf optimization: the danger is UNDER-firing (which would drop the primary liveness signal), so the matcher must never be tightened in a way that fails to match the manager in either install form — and `handleSubagentStop`'s `MANAGER_SUBAGENT_RE` filter stays as defense-in-depth even though the matcher now pre-filters.
- **The analytics/SessionStart hooks gate by WALKING UP from `cwd`, not by checking one directory.** `analytics_relay.ts` and `pipeline_ui_relay.ts` both call `resolveProjectRoot(cwd)` FIRST (which maps a git worktree to its MAIN repo + records the worktree tag), then gate with `hasPipelineDirUpTo(cwd, project_root)` — true if a `.claude/pipeline` dir exists at `cwd` or any ancestor up to (and including) the project root. This is load-bearing and depth/worktree-independent: a terminal session is still inside the project whether it sits at the root, has `cd`'d into `.claude/pipeline/<name>/steps/…` (hand-orchestrating a pipeline), or is inside a git worktree checked out under `.claude/worktrees/<name>/` (Claude Code spawns subagents there). Gating on a single `cwd/.claude/pipeline` — or even `project_root/.claude/pipeline` — silently drops every event in the nested cases. The walk is BOUNDED at the resolved project root so a stray `.claude/pipeline` far up the tree (e.g. `$HOME`) can't misclassify unrelated projects. `hasPipelineDirUpTo` is duplicated byte-for-byte in both hooks (they can't import a sibling at runtime) — keep them in sync, and never revert the gate to a single-directory `existsSync` check. Event routing stays on `resolveProjectRoot`'s git root, so a worktree's events still land in the main repo's `.runtime`, tagged with `worktree`.
- **The UI/analytics system is ON BY DEFAULT — `PIPELINE_UI_ENABLED` is the master opt-OUT switch.** Every UI hook runs at entry UNLESS `PIPELINE_UI_ENABLED` is explicitly set to a falsy value (`0`/`false`/`no`/`off` disables; unset/empty — and any other value — enables). The gate is checked as the FIRST statement of `main()` in both `analytics_relay.ts` and `pipeline_ui_relay.ts` (before `process.cwd()`, the project-root walk, or any stdin read) and at the top of `runUi` in `apps/pipeline-cli/src/commands/ui.ts` (so `/pipeline:ui` / `pipeline ui` refuse to spawn the daemon and instead print that it was opted out + how to re-enable). When opted out: no events, no mirror bindings, no `session.opened`, no daemon spawn/reconcile. The Bun process for a registered hook STILL launches (the registration lives in `hooks.json`; an env var cannot un-register it) but exits immediately — so the cost per hook call drops to ~Bun-startup only; to eliminate the spawn entirely the user must disable the plugin. The `pipelineUiEnabled()` / `uiEnabled()` helper is duplicated in all three sites (hooks can't import a sibling at runtime) — keep them in sync, including the "only an explicit falsy value disables" semantics. **This enable flag is orthogonal to the host/token security gate** (`resolveHostConfig`/`checkAuth`, next section): flipping the default ON does NOT widen the bind — the daemon still binds `127.0.0.1` only, and a non-loopback `PIPELINE_UI_HOST` still requires a mandatory `PIPELINE_UI_TOKEN`. The gate does NOT cover `pipeline event` (the journal writer is cheap and is what `pipeline logs` reads, so `/pipeline:run`'s lifecycle events are journaled even when the UI is opted out). If you add a new hook entry point, add the same first-line enable check. **Opt out per-project** via `.claude/settings.json` `env`, or in the shell before launching Claude Code (hooks inherit the session environment).
- **`PIPELINE_UI_TRANSCRIPTS` is a SEPARATE opt-out that gates ONLY the transcript mirroring/fold — the UI + basic events stay on.** ON BY DEFAULT (same falsy parse as `PIPELINE_UI_ENABLED`), it is a *narrower* switch than the master one: it disables just the privacy-sensitive transcript work — (a) the daemon copying transcript CONTENT into a run's chat panel (`mirror.ts`), (b) the transcript-folded per-run token/tool analytics (`resolveRunTranscript` → `/api/run-stats` / `-failures` / `-breakdown`), and (c) the `Stop` hook's `turn.usage` transcript token tail — while leaving the daemon, `session.opened`, and the basic lifecycle events (`pipeline.*` / `iteration.*` / `tool.called` / `manager.stopped`) fully working. The gate lives at THREE seams, each of which independently prevents a transcript from being read/copied: the **binding writers** (`analytics_relay.ts appendMirrorBinding` + `pipeline event register-mirror-binding` in `event.ts`) still write the binding for run-correlation but with `transcript_path: null` when off; the **daemon** snapshots `TRANSCRIPTS_ENABLED = pipelineUiTranscriptsEnabled(process.env)` at boot (`server.ts`) and — because `resolveRunTranscript` has a `findTranscriptByRunId` SCAN fallback that would otherwise re-discover the transcript even with a null binding — short-circuits `resolveRunTranscript` to `transcriptPath:null` (no scan) AND constructs `MirrorService({ enabled: TRANSCRIPTS_ENABLED })` (which then binds/tails nothing); the **Stop hook** early-returns before tailing. The reader is duplicated in the two standalone packages that can't import `apps/pipeline-ui/lib.ts` at runtime — `hooks/analytics_relay.ts` and `apps/pipeline-cli/src/lib/event.ts` — plus the shared daemon copy in `lib.ts`; keep the three in sync. **Orthogonal to** `PIPELINE_UI_ENABLED` (master), `PIPELINE_STATS_ENABLED` (the `.stats` fold, gated in `stats_relay.ts` — this switch does NOT touch it), and the host/token binding. With it off the dashboard degrades gracefully to "a run with no transcript" (the exact same empty path a headless-only or daemon-was-off run already takes — no crash, `transcript_found:false`, drive-envelope usage still surfaces). Regression coverage: `apps/pipeline-ui/tests/transcript-opt-out.test.ts` (reader parse + hook split + `MirrorService` gate + master-off subprocess) and the `register-mirror-binding pointer gate` block in `apps/pipeline-cli/tests/event.test.ts`.
- **Per-run analytics are folded from the TRANSCRIPTS, not the hook events.** The RUN_ANALYTICS panel (per-run tools/failures/agents/tokens) is computed by `apps/pipeline-ui/transcript-stats.ts` (the manager transcript + every in-window subagent transcript, folded over the run's `[started_at, ended_at]` window), served by `GET /api/run-stats`, and consumed by the web `useRunStats` hook — which OVERRIDES the event-folded `RunState.stats` in `StatsPanel`. This exists because the hook-emitted `tool.called`/`turn.usage` are an unreliable stats source: ground-truth validation showed `turn.usage` (Stop hook, MAIN-transcript tail) never sees subagent tokens (per-run tokens came out ~0) and `tool.called` leaks ~half its events to `run_id=null`. The transcript fold reproduces ground truth exactly (validated against real runs). Resolution is a LADDER: (1) `run_id → manager transcript` via `indexRunTranscripts` over `active-mirror-bindings.jsonl`; (2) when the binding is missing OR carries `transcript_path: null` (which `pipeline event register-mirror-binding` writes whenever the caller had no `CLAUDE_SESSION_ID` — the common case in production), `findTranscriptByRunId` scans `~/.claude/projects/<encoded-root>/*.jsonl`, window-prefiltered by file times, and picks the session whose CONTENT mentions the run_id the most (the driving session embeds it many times; no content match → no transcript, never window-guessing); (3) headless drive runs fall through to `findDriveUsage`'s envelope fold. `run_id → window` via the run summary. Correctness comes from the PER-ENTRY timestamp gate (a session transcript hosts many runs); the subagent-file birthtime filter is only a read-fewer-files optimization. Caching: terminal runs indefinitely (immutable transcript), live runs on a 4 s TTL; the content-scan result is cached per run (negatives on a 15 s TTL). `GET /api/run-steps` (same caching discipline) folds per-step wall-clock timings — attempts, active duration (parked needs-input time excluded), still-open windows — from the FULL journal via `stepTimingsForRun` in `lib.ts`, backing the iteration tree's duration chips + the RUN_ANALYTICS elapsed header. **The hook events are NOT removed** — they still drive the per-iteration tree (`IterationTree`/`StepDetail`), which has NOT been migrated to the transcript fold yet (so per-iteration stats keep the legacy undercount; per-RUN does not). If you change the stats fields or the fold, keep `transcript-stats.ts`, the `/api/run-stats` handler in `server.ts`, `web/src/hooks/useRunStats.ts`, `web/src/components/StatsPanel.tsx`, `tests/transcript-stats.test.ts`, `EVENTS.md` (§ Per-run analytics), **and `apps/pipeline-cli/src/lib/vendor/transcript-walk.ts`** (a deliberate vendored copy of `foldRunStatsFromTranscript`/`collectRunToolFailures`/`transcripts.ts`'s path helpers — `apps/pipeline-cli` ships standalone to npm and can't import this sibling app at runtime, so the drive-run fold in `apps/pipeline-cli/src/lib/step-transcripts.ts` carries its own copy instead; see that vendor file's header for the full lockstep note) in sync. A run whose pipeline ran while the daemon was OFF has no bound transcript → zeroed stats (expected; nothing was mirrored). Setting `PIPELINE_UI_TRANSCRIPTS=0` (previous invariant) forces this same zeroed path deliberately: `resolveRunTranscript` returns `transcriptPath:null` up front (skipping even the scan fallback), so the fold reads nothing.
- **MirrorService scope discipline (issue #11).** The daemon's `apps/pipeline-ui/mirror.ts` only reads transcripts that were explicitly bound by a hook (PreToolUse / PostToolUse / `pipeline event register-mirror-binding`) — never blindly scans `~/.claude/projects/`. A terminal Claude session that never spawns a `pipeline-manager` or worker must NEVER have its transcript copied anywhere. The `tests/mirror-scope-discipline.test.ts` suite is the load-bearing regression test for this invariant; do not weaken it. The `MirrorService` additionally honours `PIPELINE_UI_TRANSCRIPTS` (constructor `enabled`, default = env read): when off it binds nothing and tails nothing — no transcript is copied even for a genuinely bound run.
- **Event schema is versioned.** Every event carries `schema: 1`. If you change the shape, bump the schema version AND keep the daemon backward-compatible for one version (so a v0.13 daemon can still parse v0.13 events when the user upgrades the plugin mid-run).
- **Hook-side run_id correlation has four sources, in order: explicit override → `PIPELINE_UI_RUN_ID` env → `session_id` lookup in `~/.claude/pipeline-ui/active-mirror-bindings.jsonl` → null.** The env var is the historical mechanism but is NOT propagated for Paths B/C — `/pipeline:run` exports it in a Bash subshell that never reaches Claude Code's parent process, and a terminal Path-C session never exports it at all. The session-keyed mirror-binding lookup (added in 0.26.1) is the load-bearing recovery: both `/pipeline:run` (via `pipeline event register-mirror-binding`) and the bypass-spawn PreToolUse handler write bindings keyed by session_id, so the Stop and PostToolUse hooks can recover the run_id from there. Without it, every `tool.called` and `turn.usage` event during an actively-running pipeline stamps `run_id=null` and the UI's RUN_ANALYTICS panel renders zero tools/agents/tokens. If you ever add a new event source that needs run-correlation, call `resolveRunIdFromEnvOrSession(sessionId, projectRoot)` from `hooks/analytics_relay.ts` instead of reading `PIPELINE_UI_RUN_ID` directly. Treats `PIPELINE_UI_RUN_ID=""` as unset to prevent `run_id: ""` from leaking into the journal.
- **Binding terminal-state derives from TWO sources.** `findRunIdForSession` treats a binding as terminated when either (a) an explicit `terminal` record exists for that run_id in `active-mirror-bindings.jsonl` (Path-C bypass spawns write one), OR (b) a `pipeline.completed` / `pipeline.halted` event exists for that run_id in the project's `events.jsonl`. The second source is load-bearing because chain-controller bindings (Path B) never receive a `terminal` record — `pipeline event register-mirror-binding` doesn't have a counterpart subcommand, and the daemon's `MirrorService` only marks them terminal in-memory. Without the events-jsonl check, every subsequent unrelated tool call in the same terminal session after a chain completes would re-attribute to the dead run forever. Bindings older than `BINDING_MAX_AGE_MS` (7 days) are also ignored as stale. On Windows the project_root filter compares paths case-insensitively to tolerate PowerShell-vs-Node drive-letter casing drift; on POSIX it's a strict compare. If you change the lifecycle event names or add a new "pipeline finished" event type, update `collectTerminatedRunIds` in lockstep.
- **The daemon never blocks Claude Code.** Hooks always exit 0. If the daemon is unreachable, the journal still gets written — it's the source of truth; the daemon is just a renderer.
- **Dead-run protection is liveness-based, never age-based.** A run that never emitted a terminal event folds to `running` forever; clearing it must NOT key on wall-clock age (pipelines legitimately run for hours). **THREE layers** (full detail in `EVENTS.md` § Dead-run protection; the manager-stop bullet above is the contract for layer 1):
  - **(1) Primary — event-driven `manager.stopped`.** SubagentStop emits `manager.stopped` when a run's `pipeline-manager` ends; the daemon's `sweepManagerStoppedRuns` retires a run with a `pipeline.started` + `manager.stopped` but no terminal event (`pipeline.halted`, `abandoned:true`).
  - **(2) Fallback — pid-lockfile.** `/pipeline:run` writes `<runtime>/runs/<run_id>.alive` = `{pid}` (the driving process) after `pipeline.started` and removes it on terminal (`pipeline event write-liveness`/`clear-liveness`); the daemon's `sweepProjectLiveness` retires a run whose lockfile pid is a real, dead process (`pipeline.halted`, `abandoned:true`). This MUST degrade safely: never flag a pid ≤ 1, the daemon's own pids, or a live pid. **This pid-lockfile is LOAD-BEARING — do NOT remove it.** It is the *only* detector for a HARD kill (SIGKILL / crash / power-loss never fire SubagentStop, so no `manager.stopped` arrives), AND its live-pid check (`hasLiveDriver` in `sweepManagerStoppedRuns`) is the guard that stops layer 1 from falsely retiring a Path-B supervisor that is legitimately poll-waiting on a nested blocker (it stops + later re-spawns the manager). Removing it would regress both.
  - **(3) Guaranteed manual `POST /api/runs/stop`** (+ the UI Stop buttons on run cards / the overview board) kills a daemon-launched drive child when one exists and appends `pipeline.halted` (`dismissed:true`, reason "stopped by user") in every case. `POST /api/runs/dismiss` remains as the halt-only variant. Plus daemon boot self-cleanup halts `/api/chat` runs orphaned by a restart.

  If you change the lockfile shape or the write/clear/sweep contract, update `apps/pipeline-cli/src/lib/event.ts` (the `write-liveness`/`clear-liveness` subcommands of `pipeline event`), `skills/run/SKILL.md` (the write/clear call sites — now `pipeline event`), `apps/pipeline-ui/server.ts` (`sweepProjectLiveness` + `sweepManagerStoppedRuns` + the `/api/runs` + boot wiring), and `EVENTS.md` in lockstep. Reusing `pipeline.halted` (not a new status) is deliberate — it inherits the existing fold + UI + `isActive` machinery.
- **The daemon tracks the installed plugin version, not the newest version anywhere.** "Latest" is defined as *whatever `CLAUDE_PLUGIN_ROOT` Claude Code currently resolves the plugin to* — a pinned version is authoritative, including a downgrade. The reconciliation source of truth is the SessionStart hook's `CLAUDE_PLUGIN_ROOT`, NOT a semver scan of the plugin cache. The daemon stamps the `plugin_root` it booted from into both `daemon.lock` and `/api/health`; the hook compares (normalized; case-insensitive on Windows) and POSTs `/api/restart-to` on a mismatch. Do NOT "improve" this into a newest-version-wins policy — that would silently override a deliberate user pin. The handoff ordering in `scheduleHandoff` (stop server → delete own lock → spawn successor → exit) is load-bearing: the successor's `isExistingDaemonAlive`/orphan-probe boot guard refuses to start while the predecessor still holds the port or a live lock, so releasing both BEFORE spawning is mandatory. The hook's brute-force fallback (`bruteForceRestart`) exists solely so the first upgrade *to* a reconcile-capable daemon works without a manual restart — once both ends speak `/api/restart-to`, the clean handoff path is always taken. If you change the lock shape, the `/api/health` fields, or the request/response of `/api/restart-to`, update `hooks/pipeline_ui_relay.ts` (both the lock-fast-path and the `/api/health` fallback), `apps/pipeline-ui/EVENTS.md`, and `web/src/lib/sse.ts` (the `restart` event + `restarting` connection status) in lockstep.
- **The supervisor (`supervisor.ts`, Phase 3) is plumbing, not the daemon.** The hook + `/pipeline:ui` launch the supervisor; it spawns `server.ts` as a worker, respawns it on crash (crash-loop capped at 5/60s), and on a version handoff spawns the new worker itself (so monitoring persists across upgrades). `lock.pid` is ALWAYS the worker (it serves `/api/health` + owns the port); `lock.supervisor_pid` is the manager. To STOP the daemon you must signal `supervisor_pid` (and, on Windows, the worker `pid` too, since Windows doesn't reap the child when the parent dies) — killing only the worker triggers a respawn. The worker branches on `PIPELINE_UI_SUPERVISOR_PID`: supervised → write `worker-handoff.json` + exit (supervisor respawns); unsupervised (direct `server.ts`, e.g. tests) → self-spawn the detached successor (Phase 1/2 behavior). NEVER make the supervisor own the port/lock/HTTP — that would force a much larger refactor and break the worker-level boot guard that prevents duplicate daemons. On crash the supervisor respawns the SAME `targetScript` that crashed (never silently downgrades to its own version). This is deliberately NOT zero-downtime (no socket inheritance — too risky with Bun on Windows); handoffs reuse port-reclaim for a sub-second reconnect. If you change the handoff signal (`worker-handoff.json` shape, the `worker-stop` sentinel, the supervised-vs-unsupervised branch in `scheduleHandoff`, or the lock's `supervisor_pid`), update `supervisor.ts`, `server.ts`, `hooks/pipeline_ui_relay.ts` (it launches `supervisor.ts`), `skills/ui/SKILL.md`, and `EVENTS.md` in lockstep. The supervisor's stop-vs-respawn decision is driven by sentinel FILES, not the worker exit code: a worker exit 0 is ambiguous (idle-shutdown vs an external SIGTERM), so deliberate stops (idle, `already_running`) write `worker-stop` and a handoff writes `worker-handoff.json`; a bare exit with neither is treated as a crash/external-kill and respawned. **Known transient:** the FIRST upgrade *from* a pre-supervisor version (≤0.28) runs the OLD daemon's `scheduleHandoff`, which self-spawns the new worker WITHOUT a supervisor — so crash-recovery is absent until the next cold start (idle-shutdown or a session that finds no live daemon), which launches `supervisor.ts`. This is analogous to Phase 1's pre-feature brute-force transition; do not add self-relaunch hacks to "fix" it.
- **Mid-session reconciliation (Phase 2) follows the most-recent INSTALL action, still not highest-semver.** The hook only fires at SessionStart, so the daemon also watches `~/.claude/plugins/installed_plugins.json` (the file Claude Code rewrites on every per-project install/upgrade/downgrade) and hands off to the sibling install entry whose `lastUpdated` is newest — but only if it advanced past the value captured at daemon boot (so an unrelated plugin's install doesn't bounce us) and points at a different dir. "Sibling" = an entry whose `installPath` shares the daemon's plugin-root parent dir; this is how the daemon finds other versions of *itself* without knowing its marketplace name. Versions are recorded **per project** in that file — there is NO machine-global "active version" — so "newest `lastUpdated` among siblings" is the closest correct signal and it honors downgrades. The picker is `pickNewestPluginSibling` in `lib.ts` (pure, unit-tested in `tests/version-reconcile-pick.test.ts`); the daemon's `watchInstalledPlugins`/`reconcileToNewestInstalled` wire it to `scheduleHandoff`. Do NOT switch this to a cache-dir semver scan — it would override a project's deliberate pin. `normalizePathForCompare` now lives in `lib.ts` (shared by the daemon + picker); the hook keeps its own byte-identical copy because it can't import from the app dir — keep the two in sync.
- **The at-boot version gap is surfaced, not auto-fixed: `GET /api/update-status` + `POST /api/restart`.** Phase 2 deliberately seeds its baseline to the newest `lastUpdated` at boot (only FUTURE installs trigger auto-handoff; the pre-existing gap is Phase 1's job at the next SessionStart) — so a daemon can legitimately run a stale version for hours with the newer one already installed. `computeUpdateStatus` (server.ts) reports that gap via the pure `resolvePendingUpdate` in `lib.ts` (newest complete sibling ≠ running root; baseline-free by design; unit tests live with the picker's in `tests/version-reconcile-pick.test.ts`); the web TopBar polls it every 60 s and shows an **UPDATE vX** button only when a pending update exists. `POST /api/restart` is the self-service restart: no pid/plugin_root body (unlike `/api/restart-to` — the caller is by definition addressing the daemon it wants restarted), target derived server-side (pending update, else own root — same-root re-exec is allowed here since the endpoint only fires on explicit user action), then the exact same `restart` broadcast + `scheduleHandoff` path. `pipeline ui --restart` wraps it from the terminal (waits for the successor lock/health). After ANY handoff the web hard-reloads via `useReloadOnRestart` (web `lib/sse.ts`): the successor serves a NEW dist bundle, and before this the reconnected tab kept rendering stale JS; the first healthy `/api/health` after the grace window is by construction the successor (the old worker closes the port at grace). EditorPanel's `beforeunload` guard turns that reload into a browser prompt when unsaved edits exist. E2E coverage: `tests/restart-self.test.ts`.

- **The pipelines catalog (`scanPipelines` in `lib.ts`) knows target families and configured models (0.68).** A family HUB's `targets/<name>/` sub-pipelines are cataloged as their own `PipelineInfo` entries (same contract as the launcher's `listPipelineRoots`; dot-dirs like `targets/.common/` skipped), each carrying `family_hub` (the hub's name+root) and `shared_iterations` (the hub's steps a target run chains into off-plan). Every pipeline also carries `step_models` (rel → frontmatter `model:`, own + hub steps for a target) so the tree can show the CONFIGURED model for steps that haven't run — the event-observed model still wins once a run executes the step, and `file.changed` → `/api/state` refetch keeps the configured value live after edits (editor OR external). Consumers in lockstep: `web/src/types.ts` (`PipelineInfo`), `IterationTree` (merged own+shared rows, hub-aware current-step matching, configured-model pill), `App.tsx` (`selectedPipeline` falls back to longest-`pipeline_root`-prefix match on the run's `current_iteration_path` when the name lookup misses; StepDetail fetches shared rels under the HUB's name), `web/src/lib/runs.ts` (`iterationStatsByRel`/`iterationToolStatsByRel` take an optional hub name — the hub matcher can't over-match target paths because those contain `/targets/`), and `tests/lib.test.ts` + `web/src/lib/__tests__/runs.test.ts`.
- **TOOLS/AGENTS drill-down: `GET /api/run-breakdown` (0.69).** The TOOLS and AGENTS analytics tiles open a tabbed modal (`BreakdownPanel`): per-tool aggregates (calls, fails, total/avg/max duration from `tool_use`→`tool_result` timestamp pairs) with expandable individual calls, and one row per spawned agent (type + description from the spawn's `Agent`/`Task` tool_use input, parent-view duration, and the agent's OWN token/tool fold from its matched subagent transcript — greedy start-time-proximity matching, ≤180 s; `matched:false` = unknown, never fake zeros). Tokens are per-AGENT only — a single tool call has no attributable token cost (usage is per assistant turn). Fold: `collectRunBreakdown` in `transcript-stats.ts`; endpoint shares `resolveRunTranscript` + the run-failures cache discipline.
- **Per-step reasoning EFFORT in the UI (0.69).** The catalog carries `step_efforts` + `step_permission_modes` next to `step_models` (one frontmatter read per step via `stepMetaFor`); the iteration tree shows configured-or-observed effort (`⚡level`) and permission-mode chips plus the FULL per-step token breakdown (in/out/cache R/W); the launcher exposes a default-effort row + per-step effort overrides traveling as `--default-effort`/`--effort` drive flags (`default_effort`/`effort_overrides` in the launch body); `iteration.started.resolved_effort` feeds the observed value (fold in `runs.ts` `IterationStats.resolved_effort`); `/api/chat` + `/api/chat/resume` apply the resolved effort via the Agent SDK `options.effort` (`resolveChatEffort` in `model-resolver.ts`).
- **Per-failure drill-down: `GET /api/run-failures`.** The FAIL tile in RUN_ANALYTICS opens a modal listing every window-gated `tool_result` error from the run's manager+subagent transcripts — tool name + input resolved from the preceding `tool_use` block (`collectRunToolFailures` in `transcript-stats.ts`; capped at 200, chronological, excerpts truncated). The endpoint shares `resolveRunTranscript` (the run-stats resolution ladder) so both agree on which transcript a run owns; `transcript_found:false` means "no transcript", not "no failures". Fetched on tile click only; cached like run-stats (terminal forever, live 4 s TTL). Web pieces: `StatsPanel` (`onShowFailures` — tile becomes a button when `tools_failed > 0`), `FailuresPanel` (modal; attributes each failure to a step by timestamp vs the `/api/run-steps` windows).
- **The web refreshes pipeline-derived data on `file.changed`.** `useProjectState` already refetches `/api/state`; the LaunchPanel additionally re-pulls the `/api/pipelines` plan catalog (debounced 400 ms, same-project only) so per-step models/steps track saves AND external edits without a remount. The daemon side was already correct (pipeline-tree watcher → cache invalidation + broadcast).

## Editing rules

- `apps/pipeline-ui/EVENTS.md` is the schema contract. Any change to event shape requires lockstep edits in: `apps/pipeline-cli/src/lib/event.ts`, `analytics_relay.ts`, `pipeline_ui_relay.ts`, `web/src/types.ts`, `web/src/lib/runs.ts` (client folding logic), `apps/pipeline-ui/lib.ts` (server-side fold — `RunSummaryFolder` + the per-iteration `iterationToolStatsForRun`), and the relevant `/pipeline:run` emission line in `skills/run/SKILL.md`. The server (`lib.ts`) and client (`runs.ts`) per-iteration folds are mirrors of each other — keep `iterationToolStatsForRun` in sync between them.
- **Bun is now a hard requirement for the UI feature.** The UI requires it for daemon + hooks. The event journal is plain JSONL, so it can still be tailed from a shell.
- **Pre-built `dist/` is committed.** Users get zero-install — `bun install` only required for developers rebuilding the UI. Don't add a build step to `/pipeline:ui`; it should be a launcher, not a build invoker.
- **Per-iteration token discipline still applies.** The daemon is a side observer; it must never cause the main session or any subagent to read iteration content that they wouldn't have read otherwise.
