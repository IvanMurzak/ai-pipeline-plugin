# Execution modes, models & events

> Moved verbatim from the plugin `CLAUDE.md` — load-bearing contracts, loaded on demand. Follow each bullet's lockstep rules and bump the plugin `version` in `plugin.json` on any meaningful change (see `CLAUDE.md`).

- **EVENTS schema is now v4.** v3 added `pipeline.started.default_model` (the resolved pipeline-default model, or `null` if no `model:` frontmatter was set on `PIPELINE.md`) and `iteration.started.resolved_model` (the model actually used for this iteration after step→pipeline→session resolution, or `null` if it fell through to the session default). Both values now share one WIDENED value space — an alias `haiku|sonnet|opus|fable`, a canonical `claude-*` id (verbatim), or `null` (see `ModelValue` in `EVENTS.md`). Widening the accepted values of these existing OPTIONAL string fields is NOT a `SCHEMA_VERSION` bump — the on-the-wire shape is unchanged and all readers parse it; the daemon stores/displays the raw value and never coerces a valid `claude-*` id to null. **v4 adds `step_id`** (optional kebab-case string) on `iteration.started` / `iteration.resumed` / `iteration.completed`, emitted by the `pipeline-manager` ONLY in Parallel / DAG mode for overlap-safe per-iteration analytics folding (see the DAG bullet below). All of these fields are optional — the daemon MUST still parse v1/v2/v3 events without them (no `default_model` / `resolved_model` / `step_id` keys at all). **v4 also adds new event TYPES — `worktree.created` / `worktree.finalized` / `worktree.destroyed`** (emitted by the `pipeline next` command around the worktree hooks it executes in-process in `isolation: external` runs — `worktree.created`/`worktree.destroyed` since plugin 0.54.0, previously the `pipeline-manager` emitted them by hand; `worktree.finalized` added for the opt-in mandatory finalize stage; all-optional `data`, shapes unchanged). Like `manager.stopped`, a new event TYPE is **NOT a `SCHEMA_VERSION` bump** — the daemon tolerates unknown types via the status-fold's `default:` arm, so these are additive and backward-compatible; only the web `EventType` union in `web/src/types.ts` adds the literals (mandatory). `SCHEMA_VERSION` is defined in `apps/pipeline-cli/src/lib/event.ts` (the runtime emitter `pipeline event`), `apps/pipeline-ui/server.ts`, and `hooks/analytics_relay.ts` (the web side carries `schema` as a plain number); bump all in lockstep. See `apps/pipeline-ui/EVENTS.md` for the full schema; the runtime emitter is `pipeline event` (`event.ts`). Backward-compat parsing is the load-bearing invariant; do not regress it.

- **Opt-in DAG / parallel execution (default stays SEQUENTIAL).** A pipeline runs in Parallel / DAG mode ONLY when `PIPELINE.md` carries `execution: parallel` frontmatter. **`depends-on` ALONE no longer triggers it** — a step that declares `depends-on` without `execution: parallel` runs sequentially and the `pipeline plan` CLI emits a warning. This gate is deliberate: it lets the manager decide sequential-vs-parallel from a single `PIPELINE.md` field instead of scanning every step, keeping a sequential run's start at O(1) reads. Otherwise it runs exactly as before — strictly sequential via each step-executor's `next_iteration` hint, with NO `step_id` on events. This default-sequential guarantee is non-negotiable: every pre-existing pipeline (no `execution: parallel`) behaves identically. New optional frontmatter: `PIPELINE.md` → `execution: parallel|sequential` (default sequential); `steps/NN-*.md` → `step_id: <kebab-id>` (default = filename stem) and `depends-on: [<step_id>, ...]` (default = the immediately-preceding step). In DAG mode the `pipeline-manager` drives off `pipeline next` (which internally runs `computePlan` — reading all frontmatter in a plain process, building + validating the dependency graph for cycles + dangling/duplicate ids; non-empty `errors` → `next` returns a `halt` action → the manager halts — and returning each topological layer as a `run-step` action with `concurrent: true` and each step's effective `model`/`isolation`); the manager spawns that ready set CONCURRENTLY (one `step-executor` per step, each with the `Agent` tool's `isolation: "worktree"` option), then on the follow-up `merge` action merges the returned worktree branches back SEQUENTIALLY (`git merge --no-ff` from the project root) — a merge CONFLICT halts the run (parallel steps must be independent / disjoint-footprint; a conflict is a designer error). Each isolated `step-executor` COMMITS in its worktree and reports `worktree_branch` + `worktree_path` (both `null` when run in-place / sequential). The manager cleans up merged worktrees (`git worktree remove`). **Worktree isolation is opt-out-able:** `PIPELINE.md` may set `isolation: manual` (default `worktree`) — in `manual` mode the `run-step` action's `steps[].isolation` is `null`, so the manager spawns parallel steps IN-PLACE (no `isolation: "worktree"` option) and `next` emits NO `merge` action, for pipelines that own their isolation (their own per-branch worktree + env/port scheme; the default git-only worktree isolation does not cover ports/env). The manager reads NO iteration files in either mode — `pipeline next`/`computePlan` does all the frontmatter reading and hands back one compact action. The `pipeline-designer` MAY emit these fields for genuinely independent branches (disjoint files, no ordering dependency beyond `depends-on`) but MUST keep emitting plain linear pipelines for ordinary sequential work, and MUST pair any `depends-on` with `execution: parallel`. If you change the DAG field names, the worktree report fields, the merge contract, or the plan/action JSON shape, update `apps/pipeline-cli/src/lib/plan.ts`, `apps/pipeline-cli/src/lib/next.ts`, `agents/pipeline-designer.md`, `agents/pipeline-manager.md`, `agents/step-executor.md`, and `EVENTS.md` in lockstep.

- **Opt-in EXTERNAL isolation (run-level, sequential-only) — `isolation: external`.** The third `isolation` value (beside `worktree`/`manual`, which are parallel-only) is a wholly separate, **run-level + sequential-only** mode for a consumer-provisioned worktree: ONE worktree per run, provisioned at run start (before the first step), shared by every sequential step, torn down once at run end. It exists for pipelines whose steps need project-specific provisioning the git-only worktree cannot supply (allocated ports, dev secrets, a rendered `.env`, submodule worktrees). The consumer drops two convention-path hook scripts at `<project>/.claude/pipeline/.hooks/worktree-{create,destroy}` (sibling to the pipeline folders, shared by all pipelines; the plugin only ever READS+executes them — the read-only-plugin invariant holds) and opts a pipeline in with `isolation: external` (+ optional `submodules: [..]`) frontmatter. When `state.isolation === 'external' && state.mode !== 'parallel'`, `pipeline next` emits NEW actions — `provision-worktree` at run start (from ALL three init/resume entry points, idempotent re-emit on resume) and `teardown-worktree` at run end (every terminal outcome — `completed`/`halted`/`depth-exhausted` — NOT `blocked-delegating`), implemented via new phases (`await-provision`/`await-teardown`) and a `worktree` record (`phase:"provisioned"|"torn-down"`). The `pipeline next` COMMAND executes the hook itself, in-process (`lib/hooks.ts` — env-var contract + JSON-on-stdout FROZEN — the consumer-side contract is enumerated in `docs/worktree-hook-contract.md`; a failed create surfaces as a terminal `halt` with reason `worktree-create hook failed: …`, a failed destroy as `teardown:{ok:false}` without stranding the run; `--manual-hooks` restores manager-side actuation), threads `worktree_path` + `worktree_env_file` + `external_worktree:true` onto each `run-step` as context (NOT a cwd/`isolation:"worktree"` Agent option — the step `cd`s itself), and emits `worktree.created` / `worktree.destroyed` (new event TYPES, no schema bump — see the EVENTS bullet above). **Backward-compat is byte-for-byte:** `external` makes `useWorktree` (next.ts) and the `merge` gate both false, `dispatchStep` still hardcodes `isolation:null`, and `parallel + external` is rewritten to `parallel + manual` at plan time (warning emitted) so it emits NO `provision-worktree`/`teardown-worktree`/`merge`. Every pipeline that does not set `isolation: external` is unchanged. If the hooks are missing when `external` is set, the run halts (fail loud, never silent in-place). **OPT-IN, ADDITIVE mandatory `finalize` stage (100% GENERIC — the plugin has ZERO knowledge of submodules/pointer-bumps/push or anything else finalize does).** A run OPTS IN when a `worktree-finalize.*` hook exists in the resolved hook dir (the PRIMARY trigger, resolved by the COMMAND — it owns project-root/hook-dir resolution and passes `opts.finalizeOptIn` into the engine) OR `PIPELINE.md` sets `finalize: true` (parsed in `plan.ts` → `plan.finalize`, threaded into `state.finalize` at init). When opted in, `pipeline next` inserts a `finalize-worktree` action (new phase `await-finalize`, `worktree` record `phase:"finalized"`) at BOTH terminal seams (`retroOrTerminal` + the `await-retro` case) BEFORE the teardown check — but ONLY on a `completed` run (`shouldFinalize` = `isExternal && worktree_provisioned && status==='completed' && finalize`). New order: `last step → [retro] → finalize (MUST succeed) → teardown → done`. The COMMAND runs the consumer's `worktree-finalize.*` hook in-process (`execFinalizeHook`, `PIPELINE_WT_ACTION=finalize` + the full create-style env, STRICT must-succeed: only an explicit `{"ok":true}` passes) and emits `worktree.finalized`. UNLIKE `onTeardownPhase` (which ignores its record), `onFinalizePhase` INSPECTS `record.ok`: `ok:false` (or a missing/unconfirmed record, or a missing hook) sets `status='halted'` + `halt_reason` beginning `worktree-finalize hook failed:` so the run does NOT reach `done` and the worktree is PRESERVED (the halt then flows through the normal teardown-with-`outcome:halted` path — the consumer's outcome-aware destroy hook keeps the tree). A crash in `await-finalize` re-emits the idempotent finalize on resume (mirrors `await-teardown`). A run that never opts in never finalizes — byte-for-byte unchanged. ALSO: `onTeardownPhase` now records `teardown_ok`/`teardown_detail` (and `onFinalizePhase` records `finalize_ok`/`finalize_detail`) into state instead of silently dropping the result — the run still PROCEEDS to terminal (a leaked worktree never hangs the run), it just no longer swallows the signal. If you change the external-isolation field names, the hook env-var/JSON contract, the provision/finalize/teardown actions/phases/records, the finalize opt-in trigger, or the worktree-event shapes, update `apps/pipeline-cli/src/lib/plan.ts`, `apps/pipeline-cli/src/lib/next.ts`, `apps/pipeline-cli/src/commands/next.ts`, `apps/pipeline-cli/src/lib/hooks.ts`, `agents/pipeline-manager.md`, `agents/step-executor.md`, `agents/pipeline-designer.md`, `apps/pipeline-ui/EVENTS.md`, `apps/pipeline-ui/web/src/types.ts`, and `README.md` in lockstep.

- **One worker, model chosen per spawn.** There is a single `agents/step-executor.md` (`model: inherit`). `skills/run/SKILL.md` (the supervisor) resolves only the **pipeline-level default** from `PIPELINE.md` frontmatter and hands it to the `pipeline-manager`; the manager gets the per-iteration effective model (`per-run --model override ?? step.model ?? pipeline-default.model ?? session-default`) from each `pipeline next` `run-step` action's `steps[].model` (it no longer reads step frontmatter itself, nor calls `pipeline plan` directly — `next` computes it) and passes it to each `step-executor` spawn via the `Agent` tool's per-call `model` parameter, omitting it to inherit the session model. The OPTIONAL `model:` field stays default-inherit (omit → inherit). **Per-run, per-step overrides** (0.62.0): `/pipeline:run <path> --model <step_id>=<model> …` (or the same in natural language) pins single steps for ONE RUN without editing any file — the supervisor normalizes the pairs into the manager's `step_model_overrides` input, the manager passes them as repeated `--model` flags on its FIRST `pipeline next` call, and the CLI validates them (invalid model → warn + drop; unknown step_id → warn but keep, for off-plan steps), folds them into `steps[].model` at HIGHEST precedence (a value of `inherit` FORCES the session default), and persists them in the run's `next.json` (`state.model_overrides`) so flag-less loop calls and crash/blocker resumes keep applying them. **Off-plan (synthesized) steps resolve their model from disk** (0.62.0): when a `next_iteration`/`--start`/resume path is not enumerated in the plan — the normal case for a family TARGET-rooted run reaching hub-shared steps — the `pipeline next` COMMAND resolves that step's model as `override ?? its own frontmatter ?? its own enclosing PIPELINE.md default ?? run default` (injected into the pure engine as `opts.resolveOffPlanModel`; pure-engine callers without the seam keep legacy default-only behavior), so a hub step's pinned `model:` is no longer silently dropped; an off-plan `--start`/resume `current_path` is likewise dispatched as a synthesized step instead of being silently swapped for `steps[0]`. The effective vocabulary is now the aliases `haiku|sonnet|opus|fable`, any exact canonical Claude id (a `claude-*` string, e.g. `claude-opus-4-8`), or `inherit`/absent — the resolved value is passed STRAIGHT THROUGH to the `Agent` `model` param (the platform accepts aliases, canonical ids, and `inherit`); never translate an alias to a canonical id before spawning, and never coerce a valid `claude-*` id to null. The former `pipeline-executor-haiku` / `-sonnet` / `-opus` variant agents were **removed** once the `Agent` tool gained a per-call `model` parameter (Claude Code v2.1.x) — do not reintroduce them, and do not reintroduce a "keep four bodies byte-identical" invariant or a `subagent_type`-by-tier dispatch table. (The worker was renamed `pipeline-executor` → `step-executor` in the Phase-2 UI rekey; the hook detector still tolerates the legacy name.)

- **Per-step reasoning EFFORT is the `model:` twin (0.69).** Steps and `PIPELINE.md` may carry an OPTIONAL `effort:` frontmatter field (vocabulary `low|medium|high|xhigh|max|inherit`; absent → inherit) resolving through the identical ladder inside the CLI: `per-run --effort override ?? step.effort ?? pipeline-default.effort ?? session default` — `normalizeEffort` in `plan.ts` validates, `steps[].effort` carries the resolved value on every `run-step` action, `next.json` persists `effort_overrides`, off-plan steps resolve via `opts.resolveOffPlanEffort`, and the auto-emitted `iteration.started` stamps `resolved_effort` (values-only EVENTS.md addition, schema stays 4). **Application differs by runner** (verified platform facts, 2026-07): the HEADLESS runner (`pipeline drive`) applies it for real — the executor template carries `--effort {effort}` and re-passes it on every invocation because `claude --effort` does NOT persist across `--resume`; the daemon's Agent-SDK chat path applies it via `query({ options.effort })` (`resolveChatEffort` in `model-resolver.ts`); the MANAGER path passes it to the `Agent` tool only IF the harness exposes a per-call `effort` parameter (none does as of 2026-07 — the manager spawns without it and the step inherits the session effort; never smuggle effort through prompt text). The launcher UI exposes a default-effort row + per-step effort overrides that travel as `--default-effort`/`--effort` drive flags. Change `EFFORT_LEVELS` (plan.ts) / `EFFORT_KEYS` (web types.ts) / the model-resolver's duplicate set in lockstep if the platform vocabulary ever changes.

- **Model shorthand source of truth lives in `apps/pipeline-ui/lib.ts`.** The map `MODEL_SHORTHAND_TO_ID` (with the `ModelShorthand` type) is the single canonical translation from the aliases `haiku|sonnet|opus|fable` to the Anthropic canonical model IDs (`claude-haiku-4-5-20251001`, `claude-sonnet-4-6`, `claude-opus-4-8`, `claude-fable-5`). **Canonical `claude-*` ids are accepted directly everywhere an alias is and pass through resolution unchanged** (`isCanonicalModelId` / `canonicalModelId` / `resolveStepModel` in `lib.ts` honor them verbatim); `inherit` and unrecognized non-`claude-*` values resolve to null (session default). When Anthropic ships newer versions of any tier, that one constant is the only line to edit — the resolver (`resolveStepModel` / `resolveChatModel` in `model-resolver.ts`), the server (`/api/chat` in `server.ts`), and the writer all consume it. Do not duplicate the mapping elsewhere; if a new caller needs the IDs, import from `lib.ts`.

- **The disambiguator's model TIER is load-bearing (Haiku), not the exact version.** `pipeline-disambiguator` is pinned to `model: haiku` (the alias — it tracks the latest Haiku, no version churn) deliberately: running it on Sonnet/Opus would defeat the cost optimization that motivated the refactor. The *cheap tier* is what matters; do not move it to a pricier model without justifying it in the PR. The agent is engineered for a small, well-scoped reasoning task that fits Haiku's strengths.

- **The three "brain/code" agents are pinned to Opus + max effort.** `pipeline-designer`, `pipeline-improver`, and `pipeline-script-creator` set `model: opus` (the alias — tracks the latest Opus, no version churn) + `effort: max` in their frontmatter: pipeline design, doc-improvement judgment, and extraction correctness (a wrong extraction bakes a bug into the pipeline forever) are high-leverage and infrequent, so they get the most capable model at maximum reasoning. `effort: max` is **un-overridable** (there is no per-call effort parameter). `model`, however, IS overridable by a caller's per-call `model`, so the callers MUST NOT pass one: the `pipeline-manager` (spawning improver/script-creator) and `/pipeline:design` (spawning the designer) explicitly pass NO `model`. If you add a new spawn site for any of these three, do not pass `model` there either. Use the alias `opus`, never a pinned `claude-opus-*` id, so a version bump never requires touching these files.

- **Self-improvement loop is TWO TIERS — Tier-1 (between steps) + Tier-2 (end-of-run retrospective).**
  - **Tier-1 — step-executor emits one blocking brief → `pipeline-manager` dispatches `pipeline-improver` BETWEEN steps → optionally `pipeline-script-creator`.** The step-executor includes a structured `improvement_brief` in its final report when (and only when) the iteration's own docs (not the consumer project) were flawed and it has a concrete correction it needs landed before the next step. The step-executor may also flag a script-extraction candidate (`SCRIPT-EXTRACTION CANDIDATE` bullet) inside the same brief — it does NOT extract scripts itself. `pipeline-manager` (depth 1) spawns the improver synchronously, then parses the improver's structured report for its `script_creation_briefs` LIST and spawns `pipeline-script-creator` once per entry (0 or 1 in Tier-1), all before spawning the next step-executor so the next iteration reads the fully-updated files from disk.
  - **Tier-2 — end-of-run feedback folder → ONE Opus `pipeline-improver` (batch mode) → Opus `pipeline-script-creator` per brief.** Throughout a run, each step-executor journals EVERY problem it hits (not just blocking ones) into `<pipeline_root>/.feedback/<run_id>/` — one file per problem, named `<step_id>-NN.md`, with frontmatter `category` (`doc-flaw` | `ambiguity` | `script-candidate` | `project-issue` | `env` | `friction`) + `iteration` + `step_id`, and `## Problem` / `## Evidence` / `## Suggested fix` body. The folder is gitignored via a SELF-CONTAINED stub: `<pipeline_root>/.feedback/.gitignore` containing a single `*` line (so the whole tree is ignored regardless of project gitignore setup). The `pipeline-manager` creates the dir + stub ONCE at run start (before any step spawn, so parallel spawns don't race) and passes `run_id` + `pipeline_root` to EVERY step-executor spawn. After the chain reaches `PIPELINE_COMPLETE` or halts (on `completed`/`halted`/`depth-exhausted`, NOT `blocked-delegating`), in BOTH sequential and parallel modes, the manager runs the **End-of-run Retrospective**: Globs the feedback folder (no files → no-op, default behavior unchanged), partitions problems into **DOC-ACTIONABLE** (`doc-flaw`/`ambiguity`/`script-candidate` → feed the improver) vs **HUMAN-ONLY** (`project-issue`/`env`/`friction` → summarized to the human in the Manager Final Report's `retrospective` section, never auto-improved), spawns ONE `pipeline-improver` in batch/retrospective mode (reads current doc state first so it never re-applies a Tier-1 fix; emits a `script_creation_briefs` LIST), loops `pipeline-script-creator` over that list SEQUENTIALLY, then deletes the per-run feedback folder (keeping the `.gitignore` stub). The `/pipeline:run` supervisor surfaces the `retrospective` summary to the human.
  - **`script_creation_briefs` LIST contract change.** The improver's final report now carries `script_creation_briefs` as a LIST (0..N) instead of a single `script_creation_brief`. In Tier-1 the list is 0 or 1; in Tier-2 it may have several. The manager loops `pipeline-script-creator` over the list, once per entry, sequentially. `pipeline-script-creator`'s INPUT is unchanged — it still takes ONE brief at a time; only the improver→manager hand-off became a list.
  - **Blast radius + lockstep.** The improver and script-creator's blast radius is strictly `.claude/pipeline/<name>/` (improver: docs + emitting briefs; script-creator: docs + `<pipeline-root>/scripts/`); the feedback folder is also strictly inside the consumer project's pipeline tree, never under `${CLAUDE_PLUGIN_ROOT}`. Neither touches consumer code. The cross-agent contract is load-bearing and must agree EXACTLY: step-executor's problem-file shape ↔ manager's partition/spawn + feedback-dir setup + retrospective ↔ improver's batch input + `script_creation_briefs` LIST ↔ manager's script-creator loop ↔ Manager Final Report `retrospective` section ↔ supervisor surfacing. If you change any brief/report/feedback-file format — step-executor's `improvement_brief`, the Tier-2 problem-file shape, improver's `script_creation_briefs` LIST, the Manager Final Report `retrospective` section, step-executor's `blocker_delegation`, or any structured final-report shape — change every counterpart in lockstep: step-executor, pipeline-manager, pipeline-improver, pipeline-script-creator, and `/pipeline:run`.

- **Nested-blocker loop — executor emits brief → `pipeline-manager` relays it → `/pipeline:run` supervisor files the issue, spawns the child run, waits, merges, re-invokes the manager.** Same reason: even though nested spawns are now allowed, a subagent still cannot wait hours (poll-sleep) for a PR to merge or hold a git merge loop reliably across its finite context, so the executor only prepares a brief, the manager relays it upward, and the supervisor (main session) does the spawning and waiting. The executor's responsibility is limited to stabilizing the worktree, picking a target repo and base branch, and emitting a complete `blocker_delegation` brief; everything else happens in the main session. If you change the brief shape or the caller flow, change both ends and update the "Executor Final Report" and "Nested-Blocker Flow" sections in lockstep. The brief fields + the supervisor-side flow are enumerated in `docs/nested-blocker-delegation.md`.

- **`Agent` tool availability by depth.** As of Claude Code v2.1.172 a subagent CAN spawn other subagents, up to a nesting depth of 5 (main session = depth 0, so `main → manager → step` sits at depth 2 with headroom). At the depth ceiling Claude Code **silently strips** the `Agent` tool — calls do not error, they simply have no effect — so any agent that relies on spawning must verify the spawn actually happened rather than assume it. In this plugin, `pipeline-manager` declares `Agent` for **chain orchestration** (depth 1, spawning the per-step `step-executor` plus the improver/script-creator at depth 2). `step-executor` ALSO declares `Agent`, but for **intra-step fan-out only** — a synchronous, iteration-instructed helper (e.g. a read-only `code-reviewer` at depth 3; see its "Intra-step fan-out" rules) — and must never spawn a `pipeline-manager`/`step-executor` or drive the chain. The remaining agents are pure leaves and must NOT declare `Agent` (`pipeline-improver`, `pipeline-script-creator`, `pipeline-disambiguator`). When auditing, a pure-leaf that declares `Agent` should have it removed — but `step-executor`'s grant is intentional. Because Claude Code silently strips `Agent` at the depth ceiling, any spawn (the manager's OR the step-executor's) must verify it actually happened rather than assume it.

- When updating path rules, search both agents and all skills for the old phrasing so drift doesn't creep in.

## Skill-layer architecture — the `/pipeline:run` supervisor (full original bullet)

A condensed summary lives in `CLAUDE.md` § Editing rules; this is the full contract:

- `run` is NOT a thin router — it is the **supervisor**. Claude Code now allows subagents to spawn other subagents (up to 5 levels deep, as of v2.1.172), so the chain loop no longer has to live in the main session. `/pipeline:run` (main session, depth 0) mints the run id, owns UI liveness + the mirror binding, and spawns a single **`pipeline-manager`** subagent (depth 1) that drives the whole chain — spawning a fresh `step-executor` per iteration (depth 2), dispatching `pipeline-improver` / `pipeline-script-creator` between steps, and emitting the per-iteration events. The supervisor stays at depth 0 only for what a subagent still cannot do: own a stable liveness pid, wait hours for a blocker PR to merge (the nested-blocker poll-wait), and surface results to the human. The manager returns a structured `Pipeline Manager Final Report`; the supervisor acts on it (emit `pipeline.completed`/`halted`, or run the nested-blocker flow and re-invoke the manager to resume). The manager holds only a thin cursor — the disk (iteration files + the event journal) is the source of truth, so the supervisor can re-spawn the manager fresh on crash/overflow without losing progress.
