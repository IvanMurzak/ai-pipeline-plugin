# The pipeline CLI — commands & contracts

> Moved verbatim from the plugin `CLAUDE.md` — load-bearing contracts, loaded on demand. Follow each bullet's lockstep rules and bump the plugin `version` in `plugin.json` on any meaningful change (see `CLAUDE.md`).

- **`pipeline clone <name>` scaffolds a bundled ready-made pipeline into a project — the local-first onboarding entry point.** `pipeline clone <name> [--force] [--dir <path>] [--list] [--json]` (`src/commands/clone.ts` + the `src/lib/templates.ts` registry) copies a bundled template's whole folder tree into `<cwd-or---dir>/.claude/pipeline/<name>/`. Templates SHIP INSIDE the package (`apps/pipeline-cli/templates/<name>/…`, a valid `PIPELINE.md` + `steps/` per template) and are resolved relative to the CLI's OWN source (`import.meta.dir`), never the cwd — so `clone` works identically from a plugin checkout and from an installed npm tarball (the published `bin` is `src/cli.ts` verbatim; `src/lib` → `../../templates` in both layouts). Cross-platform (`node`/`bun` `path`+`fs` only; `cpSync` recursive tree copy). It REFUSES to overwrite an existing `./.claude/pipeline/<name>/` unless `--force` (which removes-then-copies, so no stale file survives); `--list` prints the registry; an unknown name errors (exit 2) listing the available templates. Exit `0` cloned/listed · `1` target exists without `--force`, or copy failed · `2` usage. **Adding a template** (e.g. the real `support-answer` pipeline) = drop `templates/<name>/` (mirror `example-minimal`) + add a `{name, description}` entry to `TEMPLATES` in `src/lib/templates.ts`; that `.ts` edit trips CI, which runs `tests/clone.test.ts` — every registered template must plan cleanly (`computePlan` errors empty) — and `tests/packed-artifact.test.ts`, which asserts the template ships in the tarball. Contract is load-bearing across `src/commands/clone.ts`, `src/lib/templates.ts`, `tests/clone.test.ts`, and this section — change them in lockstep.

- **The `pipeline` CLI is where deterministic, LLM-free work lives.** `apps/pipeline-cli/` is a single local TypeScript app run with Bun (`bun "${CLAUDE_PLUGIN_ROOT}/apps/pipeline-cli/src/cli.ts" <command>`) — no build step for the plugin's own use, exactly like `apps/pipeline-ui/server.ts`. The principle: anything an agent would otherwise compute by reading files (enumerating steps, resolving models, building/validating a DAG, deciding what runs next) belongs here instead, so the agent reads ONE compact result and pays near-zero tokens. **Commands: `hash`, `plan`, `match`, `event`, `route`, `next`, `ui`, and `logs`.** `pipeline hash --root <pipeline_root> [--json]` computes the deterministic SHA-256 content hash of a pipeline — `PIPELINE.md` + every file under `steps/**` and `scripts/**` — and outputs `sha256:<hex>` (plain) or `{"content_hash":"sha256:<hex>"}` (`--json`). The hash is order-independent (files sorted by POSIX-relative path), rename-sensitive, and OS-stable (CRLF→LF normalized by default). This is the SAME identity the cloud registry uses (`registry/hash.ts`) and runners verify against lease pins; the runner (`c4` task) shells this same CLI to verify content hash before executing. Exit 0 on success / 2 on missing/invalid root. `pipeline plan --root <pipeline_root> [--default-model <m>] [--model <step_id>=<m> ...] [--default-effort <level>] [--effort <step_id>=<level> ...]` reads `PIPELINE.md` + every `steps/**/*.md` frontmatter and prints the execution plan JSON (`mode`, `isolation`, `default_model`, `default_effort`, ordered `steps` with effective `model` + `effort` + `depends_on`, DAG `layers`, normalized `model_overrides` + `effort_overrides`, `errors`, `warnings`; exit 1 when `errors` non-empty; repeatable `--model`/`--effort` pin single steps at HIGHEST precedence — beat the step's own frontmatter; `inherit` forces the session default; invalid value → warn + drop, unknown step_id → warn + keep for off-plan steps). **Reasoning effort (`effort:` frontmatter + the two flags) resolves through the exact same ladder as the model** — override ?? step ?? pipeline ?? null (inherit); vocabulary `low|medium|high|xhigh|max|inherit` (verified against `claude --effort`); `normalizeEffort` in `plan.ts` is the single validator — still public/library-exported, though the manager now reaches it transitively through `pipeline next`. `pipeline match --pipelines-dir <dir> (--task <t> | --issue <ref>) [--top N] [--neg-threshold N]` is the BM25 matcher both `/pipeline:find` and `/pipeline:dispatch` tier 1 call. `computePlan()` / `matchPipelines()` / `computeNext()` are library-exported (`src/index.ts`) for embedding in other local projects, and `bun run build` emits a Node-targeted `dist/cli.mjs`. The model-resolution rules in `plan.ts` MUST match the documented vocabulary (aliases `haiku|sonnet|opus|fable`, `claude-*` ids, `inherit`/absent→null) and the `MODEL_SHORTHAND_TO_ID` story in `lib.ts`; the mode gate (`execution: parallel` required, `depends-on` alone → warning) MUST match `pipeline-designer.md` + `pipeline-manager.md`. `pipeline event <type|write-liveness|clear-liveness|register-mirror-binding> [k=v…]` is the runtime UI-event emitter that `/pipeline:run` (supervisor lifecycle + liveness + mirror binding) and `pipeline-manager` (retrospective improver/script events ONLY — the main-loop per-iteration events are auto-emitted in-process by `pipeline next` via the same `emitEvent()` writer) call. `register-mirror-binding` honours `PIPELINE_UI_TRANSCRIPTS` (default ON): when off it still records the binding for run correlation but with `transcript_path: null`, so the daemon never mirrors that session's transcript — orthogonal to `PIPELINE_UI_ENABLED` and `PIPELINE_STATS_ENABLED` (see docs/ui-subsystem.md). **`pipeline route --root <pipeline_root> --run-id <id> --from <step_id> --flags <json>`** is the standalone Variant-A routing engine: for a pipeline that opts in with a `## Graph` JSON block in `PIPELINE.md` (a map of `step_id` → conditional edges `{when, goto|done, max}`), it evaluates graph + the just-finished step's result flags + this run's per-edge counters (persisted at `<pipeline_root>/.runtime/<run_id>/route.json`, gitignored) and prints the next action (`run`/`done`/`halt`). This is what makes bounded-retry loops (e.g. a `review` step looping back to `implement` up to `max: 3` times, then skipping) declarative — the counter/skip logic lives ONCE in the graph + `graph.ts`, never duplicated in step bodies. A pipeline with no `## Graph` section is untouched (legacy sequential/DAG via `next_iteration`). `computePlan()` surfaces + validates the graph (dangling/duplicate goto, bad `max` → plan errors). `pipeline route` stays public/library-exported, but inside a run the manager no longer calls it directly — `pipeline next` consumes `routeNext()` internally (see the mechanical-manager bullet). This `result_flags` + graph-routing contract is load-bearing across `step-executor.md`, `pipeline-manager.md`, `pipeline-designer.md`, and `graph.ts`/`route.ts`/`next.ts` — change them in lockstep. Lib fns are exported from `src/index.ts` for embedding.

- **Pipeline variables (`${PP_*}`) parameterize a pipeline across runs — declared once, supplied per run.** An optional `## Variables` section in `PIPELINE.md` (authoring guidance: `pipeline-designer.md`) declares `- PP_NAME (required)` / `- PP_NAME (default: ...)` bullets; iteration/manifest BODY text and a script step's `command:`/`script:` values then reference one as `${PP_NAME}`, with an optional inline fallback — `${PP_NAME:-default}` (used when the variable is unset OR empty, POSIX `:-`) or `${PP_NAME-default}` (unset only; a resolved-empty value is kept) — and `$$` escapes a literal token (`$${PP_NAME}` renders `${PP_NAME}`, never expands). `next`, `drive`, and `step run` all accept the same repeatable `--var NAME=value` (split on the FIRST `=` only, so `PP_URL=a=b` resolves to `a=b`; `NAME=` is an explicit resolved-empty value, distinct from unresolved) and `--vars-file <path>` (dotenv format: `KEY=VALUE` lines, `#` comments, optional `export ` prefix; a malformed line is a loud startup error naming the line NUMBER, never silently skipped or partially applied) — `--vars-file` entries load first, a same-named `--var` flag wins. Each declared name resolves `--var`/`--vars-file` > the operator's environment > the manifest `(default: ...)` (a `required` variable is NEVER satisfied by a default — manifest or inline — only by a supplied value); resolution is validated FAIL-FAST and AGGREGATED (never first-error-only) before any run action: a missing required variable, an unknown supplied name (a typo, or a project `.env` pointed at by mistake — non-`PP_`/undeclared entries are rejected, never bulk-imported), or any other substitution problem halts listing every problem, each variable's declaration line, and every occurrence (file:line), plus the exact remedy. On `next`/`drive` the resolved map is FROZEN into `next.json` at run init: `--resume` reuses it VERBATIM (no environment re-read — environment drift mid-run can never change a running pipeline's values), and supplying `--var`/`--vars-file` against an already-frozen run is a loud usage error (exit 2, state untouched) — start a new run to change values. A run whose saved state PREDATES the `variables` key (an older run, or the pipeline's FIRST run after gaining declarations) accepts `--var` one more time and writes the resolved map back (one-time migration, never repeated). `step run` resolves identically but NEVER freezes or persists anything (ephemeral, dry-run only — see its own bullet below). **Trust rules**: `PP_*` values are non-secret configuration BY CONTRACT — they are visible verbatim in rendered files, params files, child-script environments, logs, events, and (for agent steps) the step-executor's LLM context, where a substituted value is **untrusted data, never an authored instruction** — never design a variable to carry a secret (secret-looking names are lint-warned; scripts keep reading real secrets straight from the process environment). **Footgun**: no registry reserves the `PP_` namespace, so a `PP_*` name already set in the operator's shell/CI environment silently satisfies a declared variable via the environment tier — no flag, no prompt; check your environment (or pass an explicit `--var`, which always wins) before running an unfamiliar pipeline. Script-step argv/env/Params substitution details — including the argv[0]-substitution ban and the `.bat`/`.cmd` block — live in `docs/script-steps.md`.

- **`pipeline next` is the orchestration state machine — the `pipeline-manager` is now a thin actuator over it.** `pipeline next --root <pipeline_root> --run-id <id> [--start <iteration-path>] [--default-model <m>] [--model <step_id>=<m> ...] [--default-effort <level>] [--effort <step_id>=<level> ...] [--record '<json>' | --record-file <path>] [--manual-scripts] [--resume]` (`apps/pipeline-cli/src/lib/next.ts` + `src/commands/next.ts`) owns ALL run control flow: sequential advancement off the step's `next_iteration`, graph routing (it consumes `routeNext()` + persists the same per-edge counters in its own `next.json`, so it does NOT call the `route` command), DAG-layer stepping, the Tier-1 improver / script-creator gating, the end-of-run retrospective gate (it counts `<pipeline_root>/.feedback/<run-id>/*.md` itself), and the terminal `done`/`halt`/`blocked` decision. Each call returns ONE action — `run-step` (with `concurrent` + a `steps` list of `{step_id,path,model,isolation,index}`; for an `external`-isolation run each step also carries `worktree_path` + `worktree_env_file` + `external_worktree:true` as informational context), `merge` (worktree branches), `run-improver`, `run-script-creator` (`number`/`of`), `retrospective`, `continue` (a `type: script` call-budget boundary — see the script-step paragraph below), `done`, `halt` (`reason`+`status`), or `blocked` — and the manager performs it, then calls again with `--record '<json>'` (or `--record-file <path>`, pointing at a JSON file — for step records, the file the step-executor wrote itself at `<pipeline_root>/.runtime/<run-id>/records/<step_id>.json`, eliminating manager transcription) describing the outcome (`{kind:"step"|"layer"|"merge"|"improver"|"script"|"retro"|"worktree"|"continue", …}`; the `worktree` record carries `phase:"provisioned"|"torn-down"`, and `{kind:"continue"}` is the caller's no-op reply to a `continue` action). The engine's `provision-worktree`/`teardown-worktree` actions (external isolation) are consumed INSIDE the command by default: `commands/next.ts` executes the consumer's create/destroy hook in-process (`lib/hooks.ts`), emits `worktree.created`/`worktree.destroyed`, feeds itself the worktree record, and returns the next REAL action with top-level `provisioned:{worktree_path,branch,env_file}` / `teardown:{ok,detail}` info attached (a create failure → terminal `halt` with reason `worktree-create hook failed: …`; a destroy failure never strands the run); the `--manual-hooks` flag restores the legacy raw provision/teardown actions for a caller that actuates them by hand. **`type: script` steps are executed IN-PROCESS the same way — zero LLM tokens** (the frozen process I/O contract lives in `docs/script-steps.md`): when a `run-step` action's `steps[]` includes a `type:"script"` member, `commands/next.ts` resolves its `## Params` bindings into `<pipeline_root>/.runtime/<run-id>/params/<step_id>.json`, spawns the script through the shared `HOOK_RUNNER` wrapper (interpreter resolved by extension via `lib/hooks.ts`), takes the last-JSON-object line of stdout as the result, persists `output` to `<pipeline_root>/.runtime/<run-id>/outputs/<step_id>.json`, writes an attempt-ledger entry keyed by `(step_id, dispatch_index)` (double-execution protection across kills/crashes), synthesizes the `{kind:"step",outcome:"completed",flags,output}` record itself, and returns the next REAL action — so script chains cost nothing and the caller never sees them. A failure writes a failure record (`.runtime/<run-id>/failures/<step_id>-<dispatch_index>-<attempt>.json` + a sibling `.log`) and a `.feedback/<run-id>/` file, then follows the step's `on-failure` policy — `halt` (default) or re-dispatch the SAME step as an agent-type `run-step` carrying `fallback:'script-failure'` + `failure_record` — while `retries:` re-runs class-`transient` failures mechanically and `env` always halts. Because in-process scripts run inside the manager's 10-minute Bash call, the command enforces a soft **call budget** (`CALL_BUDGET_MS`/`SAFETY_MARGIN_MS`/`MANAGER_SAFE_TIMEOUT_S` in `lib/script-types.ts`): a script that would not fit the remaining budget yields a **`continue` action** — the caller performs nothing and immediately re-calls `pipeline next … --record '{"kind":"continue"}'` in a fresh Bash window, so a chain of script steps spans any number of calls (`pipeline drive` runs with an infinite budget — only the per-call execution cap can still yield `continue`, which drive answers itself with the same record). Script members of a concurrent DAG layer run first and are held in `state.partial_layer_results`, so the returned `run-step` carries only the agent members. `--manual-scripts` (mirrors `--manual-hooks`) restores raw script-step `run-step` actions for a caller that executes + records them by hand. The script-step record/action contract is load-bearing across `lib/script-step.ts` + `lib/script-types.ts`, `next.ts`, `commands/next.ts`, `step-schema.ts`, and `docs/script-steps.md` — change them in lockstep. The command ALSO auto-emits the per-iteration UI events (`iteration.started` from outgoing `run-step`s, `iteration.completed` from incoming step/layer records, `improver.*`/`script_creator.*` around the Tier-1 actions) — best-effort, never affecting output or exit code. State persists at `<pipeline_root>/.runtime/<run-id>/next.json` (gitignored). **Per-run step-model overrides**: repeatable `--model <step_id>=<m>` flags apply at HIGHEST precedence (beating the step's own `model:` frontmatter; `inherit` forces the session default; a malformed pair is a loud exit-2), are persisted into `next.json` (`model_overrides`) at init, and keep applying on flag-less loop calls and resumes; passing flags again replaces the persisted set. **Per-run step-effort overrides** (`--default-effort <level>` + repeatable `--effort <step_id>=<level>`) follow the identical contract — persisted as `effort_overrides`, folded into every `run-step` action's `steps[].effort`, stamped as `resolved_effort` on the auto-emitted `iteration.started`. **Pipeline variables**: repeatable `--var NAME=value` / `--vars-file <path>` resolve + FREEZE the pipeline's declared `PP_*` variables at run init (fail-fast, aggregated validation before any action; `--resume` reuses the frozen map verbatim; `--var`/`--vars-file` against an already-frozen run is a usage error, exit 2) — full contract in the "Pipeline variables" bullet above. **Off-plan model/effort resolution**: a `next_iteration`/`--start`/resume path NOT enumerated in the plan is dispatched as a synthesized step (never silently swapped for `steps[0]`) whose model/effort resolve as `override ?? its own frontmatter ?? its own enclosing PIPELINE.md default ?? run default` — the command injects the disk-reading resolvers into the pure engine via `opts.resolveOffPlanModel` / `opts.resolveOffPlanEffort` (covered by `tests/model-overrides.test.ts`). `--resume` (and any no-record call on an already-initialized run) re-enters at `--start` — covering both nested-blocker resumes and crashed-manager re-spawns; graph loop counters survive the re-entry. The manager NO LONGER decides anything about control flow, builds DAGs, routes graphs, gates the improver/retrospective, runs worktree hooks, or emits main-loop events — it spawns, parses LLM reports, runs `git merge`, runs the retrospective's batch improver work (emitting only THAT pass's improver/script events), and relays blockers (event/feedback/report SHAPES are UNCHANGED; only the emitter and hook-executor moved into code). The record/action contract is load-bearing across `next.ts`, `commands/next.ts` + `lib/hooks.ts`, `pipeline-manager.md`, and (for the report/feedback fields it parses) `step-executor.md` + `pipeline-improver.md` — change them in lockstep. `computeNext()` is exported from `src/index.ts`; the engine is exhaustively unit-tested in `tests/next.test.ts` (sequential/graph/parallel advancement, improver+script gating, retrospective gate, halt/blocked/resume, merge conflict) and the command-level hook execution + event auto-emission in `tests/hooks.test.ts`.

- **`pipeline step run` is the author's harness for a single `type: script` step — no run required.** `pipeline step run <iteration.md> [--param k=v …] [--json]` resolves the step's `## Params` (statics / `default`s; a `${steps.…}` binding REQUIRES a matching `--param` override — else exit `2` with a clear message, since there is no run to read a prior step's `output` from), executes the script exactly as the runtime would (same env / cwd / timeout machinery + the frozen `PIPELINE_STEP_*` env contract in `docs/script-steps.md`), and prints the parsed result + the would-be `{kind:"step",…}` record (`--json` for machine form). It **NEVER touches any run state** — no records, ledger, outputs store, or feedback. Exit codes: `0` the script ran and returned `ok:true`; `1` it ran but failed (`ok:false` or a failure class); `2` usage — bad args or an unresolved `${steps.…}` binding with no `--param`. This is how a pipeline author tests a script step in isolation; the full runtime path is `pipeline next` (script-step paragraph above). `--var NAME=value` / `--vars-file <path>` resolve the pipeline's declared `PP_*` variables (same precedence and fail-fast validation as a real run — full contract in the "Pipeline variables" bullet above) and substitute them into the step's `command:`/`script:` surfaces + `## Params` bindings exactly like a real run, but EPHEMERALLY: nothing is frozen, persisted, or written back — an unresolved required variable is a `2` usage error, same as an unresolved `${steps…}` binding. Contract shared with `lib/script-step.ts` / `lib/script-types.ts` + `docs/script-steps.md` — change in lockstep.

- **`pipeline submodule bump` is a GUARDED git primitive — the mistake is impossible by construction, not by care (Phase-1 of the guarded-CLI architecture).** `pipeline submodule bump --project-root <superproject> [--submodules a,b] [--base <branch>] [--source-worktree <path>] [--dry-run] [--json]` (`src/commands/submodule.ts` + `src/lib/{land,drift,git}.ts`) records superproject submodule-pointer change(s) on the base branch and pushes them **isolation-safely**: the shared checkout is NEVER `checkout`/`reset`/`switch`ed — its ONLY mutation is `fetch` + `merge --ff-only`; all branch/commit work happens in a THROWAWAY worktree off `origin/<base>`. It is the TS port of the consumer project's tested `_lib/land_to_main.py` + `_lib/submodule_drift.py` and embodies the guards whose ABSENCE caused this session's incidents: **fork-diff safety** (a pointer that differs only because base advanced past the run's fork is NOT bumped/reverted — the #132-revert fix, applied to pointers), **conflict** (a pointer base changed since the fork is skipped + surfaced, never clobbered), **reachability** (only bump to a commit reachable from the submodule's `origin/<default>` and strictly ahead of the recorded pointer), **pre-flight self-clean** (orphaned `land-*` throwaway worktrees + stale scratch branches from prior KILLED runs are reaped BEFORE anything is created, so correctness never depends on a `finally` — idempotent under a mid-run SIGKILL), and **STOP-on-error** (halt with a structured `halt_reason` + the exact manual recovery; the post-merge reconcile retries the transient index.lock/propagation-lag race with bounded backoff, then halts — never forces). It auto-detects drifted pointers from `<project-root>/.gitmodules` when `--submodules` is omitted, and `--source-worktree` reads the run's intended pointer from that worktree (its submodule checkout HEAD, or the gitlink its HEAD records) instead of the live superproject checkout. Output is ONE JSON object (`{status: committed|noop|dry-run|halted, bumped[], skipped[], pr, infra_sha, reconcile_status, merged_via_admin, halt_reason?}`); exit `0` (committed/noop/dry-run) / `1` (halted) / `2` (usage/env). **GENERIC** — no project specifics; a project with no submodules resolves an empty candidate set → noop; assumes `git`+`gh` on PATH (a missing `gh` on a non-dry-run landing is a clean exit-2 env error). Everything is expressed over injectable `GitRunner`/`GhRunner` seams (`src/lib/git.ts`) so `tests/submodule-*.test.ts` (sharded; shared world in `tests/_submodule-world.ts`) proves each guard against REAL git sandboxes (a local bare origin + a fake-gh that ff-advances it) — the isolation invariant is asserted from the git call log. `landToMain`/`classifyDrift`/`bump` are library-exported from `src/index.ts`. **A finalize hook wires it as** (paths absolute): `bun "${CLAUDE_PLUGIN_ROOT}/apps/pipeline-cli/src/cli.ts" submodule bump --project-root "$PIPELINE_WT_PROJECT_ROOT" --source-worktree "$PIPELINE_WT_WORKTREE_PATH" --base "$PIPELINE_WT_BASE_BRANCH" --json` — but the consumer finalize-hook wiring is NOT yet done (the TD does that after review). The guard-behavior contract is load-bearing across `land.ts`/`drift.ts`/`git.ts` + `tests/submodule-*.test.ts` (sharded; shared world in `tests/_submodule-world.ts`) + this README's `submodule bump` section — change them in lockstep; each guard has a named test that replays its incident.
- **`pipeline ui` is a thin launcher, NOT a daemon rewrite.** `pipeline ui [--open] [--json] [--restart]` (`src/commands/ui.ts`) detects a running dashboard daemon via `~/.claude/pipeline-ui/daemon.lock` + `/api/health`, spawns the **supervisor** (`apps/pipeline-ui/supervisor.ts`) detached when none is up (mirroring `hooks/pipeline_ui_relay.ts:spawnDaemon` — `process.execPath` when run under Bun, else a `bun` on PATH), `POST`s the cwd to `/api/register-cwd` when the project uses the plugin, and prints the URL. `--restart` POSTs `/api/restart` on a live daemon (hand off to the newest installed plugin version, else same-root re-exec) and waits for the successor lock + health before reporting; with no live daemon it degenerates to a normal start. It locates the daemon via `${CLAUDE_PLUGIN_ROOT}` or by walking up from the CLI's own dir (so it works embedded too), and requires Bun. **It must not duplicate or disturb the daemon's single-instance bind / version-reconcile / supervisor-handoff / liveness machinery** — those live in `apps/pipeline-ui/{supervisor,server}.ts` and are load-bearing. `/pipeline:ui` (`skills/ui/SKILL.md`) routes through it; the `SessionStart` hook still launches the supervisor directly (unchanged). Only the pure helpers (`parseUiArgs`, `daemonUrl`, `resolveSupervisorScript`, `hasPipelineDir`, `uiEnabled`) are unit-tested — the spawn/fetch path is integration, like the daemon-ping in `event.ts`. `runUi` honors the default-ON master switch (via `uiEnabled()`): it spawns the daemon unless `PIPELINE_UI_ENABLED` is explicitly opted out with a falsy value (`0`/`false`/`no`/`off`), in which case it prints that it was opted out + how to re-enable and exits 0 without spawning.
- **`pipeline logs` is a read-only terminal tail of the event journal.** `pipeline logs [--follow|-f] [--tail <n>] [--all] [--json] [--no-color] [--project <path>]` (`src/commands/logs.ts`) resolves `<project-root>/.claude/pipeline/.runtime/events.jsonl` (via the shared `resolveProjectRoot`, so a worktree maps to its main repo) and pretty-prints each event as a one-liner — the terminal-visible counterpart to the browser dashboard, for users who want to watch events appear without (or with a disabled) UI daemon. It is strictly READ-ONLY: it never writes the journal, spawns the daemon, or emits events, so it is intentionally NOT gated by `PIPELINE_UI_ENABLED` (that's the whole point — terminal logs even when the server is opted out). `--follow` polls the file for growth (no `fs.watch` — too flaky cross-platform, esp. Windows rotation) and handles truncation/rotation by resetting the byte offset. The pure helpers (`parseLogsArgs`, `formatEvent`, `journalPathFor`) are unit-tested; the follow loop is integration. When you add a NEW event type to `EVENTS.md`, add a `case` to `bitsForEvent` in `logs.ts` so it renders nicely (the `default` arm just dumps `data` as JSON, so an un-added type still shows — it's a nice-to-have, not a lockstep requirement).
- **`pipeline release` bumps the plugin version safely.** `pipeline release <patch|minor|major> [--plugin-root <path>] [--dry-run] [--json]` bumps the semver `version` in `.claude-plugin/plugin.json` (the field Claude Code caches plugins by) and prints the remaining release checklist (commit+push the plugin repo; bump the submodule pointer + marketplace.json version in the marketplace repo). It performs NO git operations — the bump is the only write.

- **`pipeline drive` is the EXPERIMENTAL headless runner — an entire pipeline run with NO pipeline-manager LLM agent.** `pipeline drive --root <pipeline_root> --run-id <id> --start <iteration-path> [--default-model <m>] [--model <step_id>=<m> ...] [--default-effort <level>] [--effort <step_id>=<level> ...] [--resume] [--answer <text> | --answer-file <path>] [--task <text> | --task-file <path>] [--executor-cmd <template>] [--json]` (`src/commands/drive.ts`) is the opt-in counterpart to the manager (selected per pipeline via `runner: headless` frontmatter; the command itself takes explicit args and never reads that field). `--var NAME=value` / `--vars-file <path>` forward to the same engine init `next` uses — resolved once, FROZEN into `next.json`, and a frozen resume rejects new values with a usage error (exit 2) — full contract in the "Pipeline variables" bullet above. - **`pipeline gc` is the leak janitor — verify (and, with `--clean`, reap) worktrees/branches left behind by crashed or killed runs.** `pipeline gc [--project <path>] [--clean] [--json] [--no-submodules] [--force-worktree-branches]` (`src/commands/gc.ts`; all git calls go through the injectable `GitRunner` seam in `src/lib/git.ts`) complements the runtime's own outcome-aware cleanup (the parallel-merge branch deletion and the external destroy hook's `PIPELINE_WT_DELETE_BRANCHES` — `1` only on a COMPLETED run that hasn't set `delete_branches: false` frontmatter; halted/depth-exhausted/create-failed always preserve) with an on-demand audit for the cases no runtime can cover (SIGKILL mid-run, a crashed hook, a manually deleted directory). The **report** (always printed, exit 0 even when leaks exist; exit 2 on usage / not a git repo) lists: registered git worktrees under `<project>/.claude/worktrees/` (`git worktree list --porcelain`) with each branch's merged state vs the default branch (`git merge-base --is-ancestor`; default = the `origin/HEAD` symref when resolvable, else `main`, else `master`); stale directories inside `.claude/worktrees/` NOT registered as worktrees (a dir that is a live worktree of a scanned submodule is NOT stale); prunable worktree records (`git worktree prune --dry-run -v`); and local `worktree-*` branches with no attached worktree, flagged merged/unmerged — with the explicit caveat that **squash-merged branches read as "unmerged"**, so orphaned branches are report-only and never auto-deleted unless provably merged. **Submodule scanning is ON by default** (`--no-submodules` to skip): runs provision worktrees in every declared submodule, so each crashed run historically leaked one `worktree-*` branch into EACH submodule repo (3,649 branches across ~30 submodules in the production consumer). Every initialized submodule (`git submodule status`; uninitialized ones are skipped silently and counted in `submodules_skipped`) gets the SAME scan against its OWN default branch (same ladder): orphaned `worktree-*` branches, registered worktrees whose path lies under the SUPERPROJECT's `.claude/worktrees/` (leaked registrations), and prunable records. **`--clean`** is CONSERVATIVE by construction: it runs `git worktree prune`, `git worktree remove --force`s only registered worktrees under `.claude/worktrees/` whose branch is fully merged (unmerged/detached/current ones are kept and listed with a reason), deletes the removed dirs' leftovers + other stale dirs, and `git branch -d`s (SAFE delete — never `-D`) merged branches that either match `worktree-*` or belonged to a worktree just removed; the same rules run per submodule (never the submodule's checked-out branch); it never touches the current branch/worktree and prints exactly what was kept and why. **`--force-worktree-branches`** (opt-in, requires `--clean`) additionally `git branch -D`s UNMERGED `worktree-*` branches — squash-merged run branches read as unmerged to `--is-ancestor` forever, so safe `-d` can never reap them; guard rails: only the machine-owned `worktree-*` namespace, never the current branch, and human output states how many were force-deleted per repo. `--json` emits one object `{default_branch, worktrees:[{path,branch,merged}], stale_dirs:[…], prunable:[…], branches:[{branch,merged}], submodules:[{path,default_branch,branches,worktrees,prunable}], submodules_skipped, cleaned:{pruned,removed_worktrees,kept_worktrees,removed_dirs,deleted_branches,force_deleted_branches,kept_branches,submodules:[{path,pruned,removed_worktrees,kept_worktrees,deleted_branches,force_deleted_branches,kept_branches}]}|null}` (pre-submodule fields unchanged); human mode prints a per-submodule section only for submodules with findings plus a one-line total ("N leaked branches across M submodules"), and a final "no leaks detected" line when everything is empty. - **`pipeline ci-wait` is the token-efficient GitHub CI gate — block until CI reaches a terminal state, in ONE Bash call.** `pipeline ci-wait [--pr <number|url|branch> | --branch <name> | --sha <sha>] [--repo <path>] [--timeout <sec>] [--interval <sec>] [--grace <sec>] [--fail-fast|--no-fail-fast] [--json] [--verbose]` (`src/commands/ci-wait.ts`; gh/git/clock/sleep all injectable — `tests/ci-wait.test.ts` scripts gh outputs poll-by-poll with a fake clock) exists because the LLM alternative is terrible: an agent poll loop (sleep → `gh pr checks` → read the table into context → repeat) burns a full agent turn per poll and, worse, agents have been observed waiting HOURS for full CI completion when one job already failed or hung. `ci-wait` polls `gh` IN-PROCESS, stays silent while waiting (`--verbose` opts into stderr heartbeats), and prints ONE compact result line (or one JSON object with `--json`). **Modes**: `--pr` covers a pull request via `gh pr checks --json name,state,bucket,link` (Actions + third-party checks/statuses; stdout is parsed regardless of gh's exit code, since gh exits non-zero while checks pend/fail); `--branch` resolves the branch HEAD sha ONCE at start via the API and then polls the commit check-runs endpoint for that pinned sha (a later push is a NEW gate, not a moving target); `--sha` skips resolution; NO selector = the repo's default branch (`gh api repos/{owner}/{repo} --jq .default_branch`). **Fail-fast is the DEFAULT**: the FIRST check landing in fail/cancel (or any non-passing terminal conclusion — unknown conclusions fail CLOSED) ends the wait immediately even while other jobs still run or hang; `--no-fail-fast` waits for the full picture; `--timeout` (default 1800s) caps stuck-forever CI and `--grace` (default 120s) bounds the "CI never started" case. **Exit codes**: `0` all passed (pass/skipping buckets; success/skipped/neutral conclusions) / `1` a check failed / `2` usage or gh missing/unauthenticated or unresolvable branch / `3` timeout (result lists the still-pending names) / `4` no checks appeared within the grace period — deliberately DISTINCT from success so a caller can never mistake "CI never started" for a green gate (an empty check list is grace territory, not a pass). The JSON result is `{status: success|failure|timeout|no-checks, mode: pr|commit, ref, sha, elapsed_s, polls, total, passed, failed, pending, failed_checks:[{name,state,link}], pending_checks:[names], detail}`. Intended use in iteration steps: a CI-gating step runs `bun "${CLAUDE_PLUGIN_ROOT}/apps/pipeline-cli/src/cli.ts" ci-wait --pr <n> --json` in a single Bash call (with the Bash timeout ≥ the `--timeout`) and branches on the exit code — no hand-rolled poll loops in step docs.
